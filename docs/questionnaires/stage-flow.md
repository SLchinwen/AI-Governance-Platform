## 二、填答階段詳細流程

### Stage 1：探索期（Discovery）— 由 PM 主導

```
目標：確認專案的基礎條件與範圍
```

| 填答內容 | 分類 | 題數 | 門檻 |
|---------|------|------|------|
| 專案名稱、類型、範圍 | project_basic | 8 題 | 100% 必填完成 |

**完成條件：**
- [ ] 所有 project_basic 必填題已回答
- [ ] 專案類型已確認（決定後續載入哪些題組）
- [ ] 資料敏感度已確認（決定安全要求層級）
- [ ] PM 已確認所有答案

**產出：**
- 專案 profile 初稿
- 後續題組載入建議

**品質門檻（Gate 0）：**
- Readiness Score ≥ 20（僅計算 project_basic 分類）
- 無 Error

---

### Stage 2：規劃期（Planning）— 由 Architect 主導

```
目標：定義架構方向與 AI 協作策略
```

| 填答內容 | 分類 | 題數 | 門檻 |
|---------|------|------|------|
| 架構風格、分層、部署平台 | architecture | 8 題 | 80% 必填完成 |
| AI 工具、範圍、審查流程 | ai_collaboration | 9 題 | 80% 必填完成 |

**完成條件：**
- [ ] 架構風格已確認
- [ ] 部署平台已確認
- [ ] AI 工具與使用範圍已確認
- [ ] AI 禁止範圍（紅線）已明確定義
- [ ] AI 審查流程已確認
- [ ] Architect 已審核所有答案

**產出：**
- 架構方向草案
- AI 協作規範草案
- 後續設計期的題目範圍確認

**品質門檻（Gate 1）：**
- Readiness Score ≥ 40
- architecture + ai_collaboration 分類完成度 ≥ 80%
- 無 critical 等級題目未填

---

### Stage 3：設計期（Design）— 各技術角色分別填寫

```
目標：細化技術選型與工程規格
```

| 填答內容 | 分類 | 題數 | 負責角色 |
|---------|------|------|---------|
| 後端技術棧 | backend | 9 題 | Backend Lead |
| 前端技術棧 | frontend | 9 題 | Frontend Lead |
| 資料庫設計 | database | 6 題 | Backend Lead |
| API 設計 | api | 7 題 | Backend Lead |
| 資安與合規 | security | 6 題 | Security Lead |
| 部署與維運 | deployment | 8 題 | DevOps |

**完成條件：**
- [ ] 每個分類的必填題完成度 ≥ 80%
- [ ] 所有 critical 等級題目已填答
- [ ] 條件觸發的子題已正確填答
- [ ] 各角色已確認自己負責的答案

**品質門檻（Gate 2）：**
- Readiness Score ≥ 70
- 所有 Error 等級驗證規則通過
- Warning 數量 ≤ 5
- 每個分類至少有一位角色完成填答

---

### Stage 4：驗證期（Validation）— 系統自動 + 人工確認

```
目標：確保答案完整、一致、無衝突
```

| 驗證類型 | 規則數 | 說明 |
|---------|--------|------|
| 必填檢查 | 5 條 | 條件必填題是否正確填答 |
| 交叉驗證 | 13 條 | 跨分類答案一致性 |
| 衝突檢查 | 2 條 | 答案間的邏輯衝突 |

**系統自動執行：**
- 執行 20 條驗證規則
- 計算 Readiness Score（9 維度加權）
- 產出驗證報告（Blockers + Warnings）
- 標記低信心答案

**人工確認：**
- 架構師檢視驗證報告
- 處理 Warning 項目
- 確認低信心答案是否需要修正
- 決定是否需要跨角色會議

**品質門檻（Gate 3）：**
- 0 個 Error（Blocker）
- Warning ≤ 3 且有處理計畫
- Readiness Score ≥ 70
- AI 協作分類 Score ≥ 80

---

### Stage 5：審核期（Review）— 架構師 + PM

```
目標：正式審核通過，鎖定版本
```

**審核清單：**
- [ ] 所有答案已由對應角色填寫
- [ ] 驗證規則全部通過
- [ ] Readiness Score 達標
- [ ] AI 安全邊界已確認
- [ ] SSOT 文件清單已確認
- [ ] 無未解決的退回修正

**審核流程：**
```
架構師審核 → PM 確認 → 簽核 → 鎖定版本
    │            │
    ▼            ▼
  退回修正    需要討論
  (附建議)   (安排會議)
```

---

### Stage 6：產出期（Output）— 系統自動

```
目標：產出正式文件資產
```

| 產出 | 格式 | 用途 |
|------|------|------|
| `tech-stack.json` | JSON | 技術棧結構化定義（SSOT） |
| `ai-context.md` | Markdown | AI 開發上下文 |
| `readiness-report.md` | Markdown | 就緒度評估報告 |
| `adr-draft.md` | Markdown | 架構決策紀錄草稿 |
| `cursor-rules-draft.mdc` | MDC | Cursor IDE 規則草案 |

---

### Stage 7：發布期（Publish）— DevOps + 架構師

```
目標：正式發布到 GitHub Repo
```

**發布流程：**
```
產出文件 → 建立 PR → 審核 → 合併 → 標記版本
             │
             ├── 附上 AI 使用揭露
             ├── 附上變更摘要
             └── 指定 Reviewer
```

## 三、階段切換條件

| 從 | 到 | 條件 |
|----|----|------|
| 探索期 → 規劃期 | PM 確認基本資訊 + Gate 0 通過 |
| 規劃期 → 設計期 | 架構方向確認 + AI 規範確認 + Gate 1 通過 |
| 設計期 → 驗證期 | 所有分類填答完成度 ≥ 80% + Gate 2 通過 |
| 驗證期 → 審核期 | 驗證規則 0 Error + Gate 3 通過 |
| 審核期 → 產出期 | 架構師 + PM 簽核通過 |
| 產出期 → 發布期 | 產出文件 PR 審核通過 |

## 四、退回與重新流程

### 退回觸發條件

- 審核者發現答案不正確或不完整
- 新的需求導致已回答題目需要修改
- 架構方向發生重大變更

### 退回範圍

| 退回類型 | 影響範圍 | 處理方式 |
|---------|---------|---------|
| 單題退回 | 僅影響該題 | 修改後重新提交 |
| 分類退回 | 整個分類需重新檢視 | 角色代表重新確認 |
| 全面退回 | 架構方向重大變更 | 從規劃期重新開始 |

### 退回後的版本處理

- 退回不刪除原答案，而是建立新版本
- 原版本標記為 `superseded`
- 新版本繼承未變更的答案

## 五、持續優化機制

### 每個專案完成後

1. 收集填答者回饋（難度、耗時、遺漏）
2. 分析驗證規則命中率（哪些規則最常觸發）
3. 檢視 Readiness Score 與實際 AI 協作品質的相關性
4. 識別需要新增/修改/刪除的題目

### 每季度

1. 題庫內容審核（是否需要更新選項）
2. 驗證規則有效性評估
3. 計分權重校準
4. 新增技術趨勢相關題目
5. 發布題庫新版本

### 年度

1. 全面題庫結構檢視
2. AI 使用政策更新
3. 治理流程效率評估
4. 長期知識庫價值評估
