# S1 探索期問券：白話引導手冊 v1.0

> 適用角色：PM / 顧問  
> 使用時機：專案啟動第一階段（Discovery）  
> 目標：在最短時間收齊後續技術規劃一定會用到的基礎資訊。

---

## 1. S1 到底在收什麼？

S1 不是要做技術設計，而是先回答三件事：

1. 這是什麼專案？（名稱、類型、是否新專案）
2. 規模多大？（範圍、時程、人力）
3. 資料風險多高？（資料敏感度）

只要這三件事清楚，後續問券題組才會載入正確，AI 建議才會貼近實際。

---

## 2. S1 主要資訊方向（對應後續用途）

| S1 資訊 | 為何要收 | 影響後續哪裡 |
| ---- | ---- | ---- |
| 專案名稱 | 決定所有輸出文件與 Repo 標識 | `tech-stack.json`、`ai-context.md` |
| 專案類型 | 決定要載入哪些題組 | architecture/backend/frontend/database/api/security/deployment |
| 專案範圍等級 | 估算複雜度與投入方式 | readiness 評分、排程與資源規劃 |
| 是否全新專案 | 決定是否需考慮舊系統相容 | 技術選型、遷移/重構策略 |
| 預計時程 | 決定交付節奏與方案取捨 | 里程碑、MVP 範圍 |
| 團隊規模 | 決定技術與流程重量 | 工具鏈、分工與審核深度 |
| 主要利害關係人 | 明確誰決策、誰驗收 | 審核流程、需求確認責任 |
| 資料敏感度 | 決定安全與 AI 邊界 | 合規策略、AI 可用資料範圍 |

---

## 3. PM 白話問法（可直接對客戶使用）

## 問題 1：這個案子你們想解決什麼？

- 白話問法：  
  「這次專案最想先解決哪 1-2 個痛點？」
- 你要填到哪裡：專案名稱、專案類型、非目標

## 問題 2：大概多大、多久要上？

- 白話問法：  
  「這次希望多久內上線？大概會有多少功能範圍？」
- 你要填到哪裡：範圍等級、時程、團隊規模

## 問題 3：會碰到什麼資料？

- 白話問法：  
  「這個系統會不會碰到客戶個資、交易資料或機密資料？」
- 你要填到哪裡：資料敏感度等級

## 問題 4：是全新做，還是改現有系統？

- 白話問法：  
  「這是全新系統，還是會沿用你們現在的系統？」
- 你要填到哪裡：是否全新專案（greenfield/extension/migration/refactor）

## 問題 5：誰說了算、誰最後驗收？

- 白話問法：  
  「這個案子誰會做最後決定？誰會驗收？」
- 你要填到哪裡：主要利害關係人

---

## 4. 快速填寫順序（建議 10-15 分鐘）

1. 先填：專案名稱、專案類型、是否新案
2. 再填：範圍等級、時程、團隊規模
3. 最後填：資料敏感度、利害關係人

如果資訊不完整：

- 先選最接近選項
- 在備註標 `待確認`
- 指派確認人與確認日期

---

## 5. 常見填錯與修正方式

| 常見錯誤 | 影響 | 修正方式 |
| ---- | ---- | ---- |
| 專案類型選錯 | 後續題組載入錯誤 | 先看主要交付：Web 畫面為主或 API 為主 |
| 低估資料敏感度 | AI 邊界與安全策略失真 | 有個資就至少 internal/confidential 起跳 |
| 時程亂填 | 後續方案建議不實際 | 用「目標上線日 + MVP 日」雙日期方式 |
| 直接留空 | 後續規劃卡住 | 先填待確認與 owner，不可無主資料 |
