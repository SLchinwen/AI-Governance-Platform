# 公司開發基線（SystemLead Technologies）v1.0

> 本文件定義公司目前軟體開發與 AI 協作導入的基線狀態，作為後續治理與工程推進的依據。
> 最後更新：2026-03-13

---

## 1. 公司與團隊概況

- 公司：SystemLead Technologies Co.,Ltd.（矽聯科技股份有限公司）
- 團隊定位：軟體開發與專案交付
- 目前軟體開發人力：7 人

角色配置：

| 角色 | 人數 | 主要職責 | 主要工具 |
| ---- | ---- | -------- | -------- |
| PM / 顧問 | 2 | 前端需求訪談、背景盤點、需求與目標定義、問題梳理、PDR | Cursor AI |
| 系統工程師 | 1 | Azure 雲端環境建置管理、資料庫監控、資安協作 | Visual Studio |
| 後端工程師 | 2 | .NET 後端開發 | Visual Studio |
| 前端工程師 | 2 | 前端 UI 規劃與開發 | Visual Studio |

---

## 2. 產品與服務範圍

目前主要服務：

1. 電子發票加值中心服務
2. 公益社團網站服務
3. 客製化專案開發服務

治理導入目標：

- 建立可重複使用的知識庫與問卷機制
- 支援跨角色協作並提升需求到開發的一致性
- 導入 AI 協作流程，提升交付效率與可追溯性

---

## 3. 技術與基礎設施基線

### 3.1 開發與協作工具

- IDE：Visual Studio、Cursor（依需求可補充 VS Code Remote）
- 版本控制：GitHub
- AI 協作：Cursor AI

### 3.2 雲端與資料庫

- 雲端平台：Microsoft Azure
- 目標資料庫：Azure SQL Database
- 資安遵循：ISO/IEC 27001 政策（公司內規）

### 3.3 開發伺服器方向

依既有規劃文件 `ai_dev_server_plan (1).md`，導入方向如下：

- Azure Dev Server 作為 AI 協作與整合開發環境
- 支援 .NET SDK、Node.js、Docker、Remote SSH
- 舊專案可維持本機開發；新專案優先採用 AI workflow

---

## 4. 專案治理定位

本專案屬於「治理即工程（Governance as Engineering）」範疇，目標為：

- 把治理規則文件化（policy as docs）
- 把流程工具化（policy as workflow）
- 把版本演進可追蹤化（policy as versioned assets）

治理重點：

- AI 使用邊界與揭露
- 問卷與技術規格標準化
- 審核流程與品質門檻
- 文件、規則、輸出的版本治理

---

## 5. 已知限制與風險

- 團隊規模小，需避免過重流程導致交付速度下降
- 多產品線並行，需保持標準一致但允許專案差異
- AI 導入初期可能出現輸出品質波動，必須搭配審核機制
- 合規與資安要求不可因效率而降級

---

## 6. v1.0 啟動原則

1. 先有最小可執行標準，再逐步擴充
2. 先在 1-2 個專案試行，再全團隊推廣
3. 每個流程都要有 owner、門檻、指標
4. 每次版本更新都要有變更紀錄與回溯依據
