## 一、專案背景

公司在導入 AI 協作於需求分析、技術設計、程式開發、測試與維運的過程中，逐漸發現以下關鍵問題：

### 目前痛點

| # | 痛點 | 影響 | 嚴重度 |
|---|------|------|--------|
| 1 | 每個專案的技術定義方式不一致 | 規格品質參差，跨專案經驗難以傳承 | 🔴 高 |
| 2 | 關鍵資訊分散在 PM、SA、前端、後端、維運、架構人員手上 | 資訊不對稱，決策依據不完整 | 🔴 高 |
| 3 | 工程師常靠經驗或口頭補充，缺少結構化、可稽核的技術決策 | 技術債累積，決策無法回溯 | 🟡 中高 |
| 4 | AI 協作時上下文不完整，導致產出品質不穩、返工率高 | 開發效率下降，團隊信心不足 | 🔴 高 |
| 5 | 同類型專案反覆重做相似決策，無法累積成公司資產 | 知識浪費，啟動成本重複 | 🟡 中 |
| 6 | 缺乏正式治理，難以區分「正式規格」與「說明文件」 | SSOT 模糊，版本混亂 | 🔴 高 |

### 既有基礎

公司內部已有以下治理觀念與基礎設施：

- **AI 定位明確**：AI 為輔助起草者，非事實來源（SSOT）
- **三層流程**：標準化 → 訓練 → 導入
- **Metadata 驅動**：以結構化欄位為單一真實來源，非依賴檔名與資料夾
- **階段化分工**：不同階段有對應角色、工具與產出物
- **文件治理方向**：AI 永遠讀最新、可稽核、可控、可擴充，以 GitHub Repo 最新內容為準

因此，本專案不是從零開始，而是**將既有治理觀念產品化、流程化、資料化**。

## 二、專案願景

### 定位

**AI 開發治理中台 / 技術規範知識中台**

### 願景描述

建立一套以 GitHub 為治理核心、以問卷與標準題庫為輸入、以 SSOT 與 AI 規則為輸出的公司級平台。讓每個專案都可以：

- 用**一致的方法**定義技術棧與工程邊界
- 用**一致的格式**沉澱規格與決策
- 用**一致的規則**供 AI 閱讀與協作
- 用**一致的品質門檻**決定是否可進入 AI 輔助開發

### 核心價值主張

> 用 Metadata 問卷驅動技術規範，並把結果沉澱成 AI 可讀、可控、可審核的專案資產。

## 三、核心目標

### 3.1 業務目標

| 目標 | 衡量指標 | 目標值 |
|------|---------|--------|
| 降低專案技術規範定義的不一致性 | 專案技術定義覆蓋率 | ≥ 90% 分類完成填答 |
| 降低 AI 協作因上下文不足造成的返工 | AI 產出返工率 | 降低 40% |
| 提高新專案啟動速度 | 技術規範完成時間 | 從 2 週降至 3 天 |
| 將技術決策沉澱為可重複使用的公司資產 | 知識庫累積量 | 每季 +5 個專案資產 |

### 3.2 管理目標

| 目標 | 實現方式 |
|------|---------|
| 建立可追溯的技術決策流程 | 問卷填答 + 版本紀錄 + 變更比較 |
| 建立角色分工明確的填答與審核機制 | RACI 矩陣 + 角色型題目分派 |
| 建立 SSOT 與非 SSOT 的邊界 | 文件分級制度 + 明確標記 |
| 建立可審核的 AI 使用治理 | AI 使用揭露 + 審查流程 + 安全邊界 |

### 3.3 工程目標

| 目標 | 產出 |
|------|------|
| 每個新專案產出標準化技術棧 | `tech-stack.json` |
| 自動生成 AI 友善的上下文 | `ai-context.md`、`.cursor/rules/` |
| 建立 readiness score | `readiness-report.md` |
| 結果納入 GitHub 版本治理 | PR 審查 + 版本控管 |
| 自動產出架構決策紀錄 | `adr-draft.md` |

## 五、整體流程骨架

```
Phase 1          Phase 2          Phase 3          Phase 4          Phase 5
建立標準題庫  →  專案填答流程  →  驗證與品質門檻 →  輸出機制      →  GitHub 治理
                                                                    
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────────┐   ┌──────────┐
│ 定義問題   │   │ 建立實例  │   │ 缺題檢查  │   │ tech-stack   │   │ PR 變更   │
│ 分類歸屬   │   │ 角色分派  │   │ 衝突檢測  │   │ ai-context   │   │ Reviewer  │
│ 條件邏輯   │   │ 階段填答  │   │ 品質評分  │   │ readiness    │   │ AI 揭露   │
│ 影響分析   │   │ 審核流程  │   │ 人工確認  │   │ ADR 草案     │   │ SSOT 區分 │
│ 品質權重   │   │ 版本紀錄  │   │ 門檻判定  │   │ cursor rules │   │ 版本標記  │
└──────────┘   └──────────┘   └──────────┘   └──────────────┘   └──────────┘
```

### 各 Phase 詳細說明

#### Phase 1：建立公司標準題庫
- 定義問題內容、分類、歸屬角色
- 設計條件邏輯（depends_on / triggers）
- 評估品質影響等級（critical / high / medium / low）
- 設定輸出映射（output_mapping）

#### Phase 2：建立專案填答流程
- 每個專案建立一份問卷實例
- 依角色分派題目
- 支援階段化填寫（探索期 → 規劃期 → 設計期）
- 記錄填答者、時間、信心度

#### Phase 3：建立驗證與品質門檻
- 自動執行必填檢查、衝突檢測、交叉驗證
- 計算 Readiness Score（0-100 分）
- 識別 Blockers 與 Warnings
- 決定是否可進入 AI 協作開發

#### Phase 4：建立輸出機制
- 依答案自動產出標準化文件
- 支援 JSON、Markdown 多種格式
- 輸出映射可配置、可擴充

#### Phase 5：建立 GitHub 治理流程
- 所有正式產出透過 PR 變更
- 有 Reviewer 機制
- 有 AI 使用揭露
- 有 SSOT / 非 SSOT 區分
- AI 產出必須經人工審核

## 六、先期收集資訊框架

### A. 公司級治理資料（用來定題庫）

| 收集項目 | 用途 | 來源 |
|---------|------|------|
| 常見專案類型 | 決定題庫範本 | PM 主管 |
| 常見技術棧 | 決定選項清單 | 架構師 |
| 最常反覆討論的決策 | 識別高價值題目 | 資深工程師 |
| 最常出錯的地方 | 設計驗證規則 | QA / 維運 |
| AI 協作時最常缺什麼 | 設計 AI Context 輸出 | 使用 AI 的開發者 |
| 全公司共通規則 | 定義通用規則集 | 架構委員會 |

### B. 專案級基本資料（每個新專案的起點）

- 專案名稱 / 模組 / 客戶
- 新建、延伸或整合
- 預計使用者與功能範圍
- 已知限制與既有系統依賴
- 預期時程與團隊組成

### C. 技術定義資料（問卷主體 - 9 大分類）

已建立完整題庫，共 70 題，涵蓋：前端、後端、DB、API、安全、部署、架構、AI 協作、專案基本

### D. 治理資料（讓成果可長期維護）

- SSOT 與非 SSOT 文件區分
- 變更核准權限
- 版本規則
- PR 檢查規則
- AI 使用揭露方式

## 八、風險與因應

| 風險 | 可能性 | 影響 | 因應措施 |
|------|--------|------|---------|
| 題目不符合實際需求 | 中 | 高 | 每輪由實際角色審核 + 試跑回饋 |
| 團隊抗拒填答 | 中 | 高 | 先用 1 個專案試跑、展示價值 |
| 過度設計導致進度延遲 | 中 | 中 | MVP 先做核心 3 種輸出 |
| AI 技術快速變化 | 高 | 中 | 題庫與規則設計為可版本化、可擴充 |
| SSOT 定義不被遵守 | 中 | 高 | 透過 PR 流程與審核制度強制執行 |
