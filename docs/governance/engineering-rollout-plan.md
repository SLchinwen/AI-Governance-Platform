# 工程推進計畫（AI 協作與治理導入）v1.0

> 適用對象：SystemLead Technologies 目前 7 人開發團隊
> 目標：在不影響既有交付前提下，逐步導入 AI 協作與治理標準
> 最後更新：2026-03-13

---

## 1. 推進原則

- 小步快跑：先導入最小可行流程，再以版本擴充
- 雙軌並行：知識庫建置與工程流程同步推進
- 不破壞現況：舊專案維持既有節奏，新專案優先套用新流程
- 可量測：每階段皆定義 KPI 與退出條件

---

## 2. 四階段推進（12 週）

## Phase A（第 1-2 週）：基線鎖定

交付項目：

- 完成公司開發基線文件
- 鎖定 v1.0 最小治理範圍（AI、審核、PR、版本）
- 建立專案啟動資料收集表

完成條件：

- 所有角色對責任邊界達成共識
- v1.0 文件 owner 與審核者指派完成

## Phase B（第 3-6 週）：流程最小化落地

交付項目：

- 啟用 PR 模板與 branch protection
- 啟用問卷填答與審核流程（至少 1 個專案）
- 建立 AI 使用揭露與審核清單

完成條件：

- 至少 1 個專案完成從填答到輸出的全流程
- 關鍵文件與輸出可追溯到同一版本

## Phase C（第 7-9 週）：工具化與自動化

交付項目：

- 將治理檢核納入 CI（schema、規則、文件一致性）
- 建立 release note/changelog 流程
- 建立治理文件版本標記機制（SemVer）

完成條件：

- 每次治理變更可自動檢查並生成變更記錄
- 重大變更具備明確升版依據

## Phase D（第 10-12 週）：組織化擴展

交付項目：

- 將流程擴到第 2 個專案
- 定義季度檢視會議與改善節奏
- 形成 v1.1 升版候選清單

完成條件：

- 流程穩定且不明顯拖慢交付速度
- 團隊可獨立維運治理文件與流程

---

## 3. 每週執行節奏

- 週一：規劃與風險盤點（PM + 架構師）
- 週三：中段同步（工程師 + PM）
- 週五：審核與版本更新（治理 owner）

每週固定輸出：

1. 本週變更摘要（what changed）
2. 影響評估（impact）
3. 下週待辦（next actions）

---

## 4. KPI（v1.0 建議）

| 指標 | 目標值 | 說明 |
| ---- | ------ | ---- |
| PR 首次通過率 | >= 70% | 反映流程清晰與規範可執行度 |
| 平均 PR lead time | <= 2 個工作日 | 反映審核效率 |
| 問卷必填完成率 | 100% | 確保基礎資料完整 |
| AI 協作揭露率 | 100% | 確保可追溯 |
| 治理文件更新滯後 | <= 5 個工作日 | 反映文件同步能力 |
| 重大缺陷外洩數 | 0 | 品質底線 |

---

## 5. 角色分工（RACI）

| 工作項目 | PM/顧問 | 系統工程師 | 後端 | 前端 | 架構/治理 owner |
| -------- | :------: | :---------: | :----: | :----: | :---------------: |
| 需求與目標定義 | R | I | C | C | A |

| .NET API 標準落地 | I | C | R | I | A |
| 前端標準落地 | C | I | I | R | A |
| PR/Review 流程治理 | C | C | C | C | A/R |
| AI 使用與揭露審核 | C | C | C | C | A/R |
| 文件版本維護 | C | I | I | I | A/R |

---

## 6. 風險控管與應對

| 風險 | 觸發訊號 | 應對策略 |
| ---- | -------- | -------- |
| 流程過重 | 開發 lead time 明顯上升 | 降低必填項與審核層級，改採分級門檻 |
| AI 產出品質不穩 | 修正工時偏高、回退頻繁 | 提高 AI 產出審核等級，建立 prompt 範本 |
| 文件與實作脫鉤 | 文件更新延遲 | 將文件更新設為 PR 必檢項 |
| 安全風險 | 機敏資訊進入 prompt | 強制 AI 使用清單與審核紀錄 |

---

## 7. 版本演進策略（連動治理版本）

- v1.x：流程優化、模板調整、檢核補強（向後相容）
- v2.0：治理結構重構、流程邊界重大改動（可能不相容）
- 重大版本升級需附 migration 指引與試行報告
